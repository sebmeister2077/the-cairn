# Plan: TOPS Map Game-Version Overlay

Adds a per-area Vintage Story game version layer on top of the TOPS map. Each new approved contribution stamps a version (read from a server-side env var) onto the area it touches; admins and users can toggle a tinted overlay to see which parts of the world were mapped under which game version.

This is a standalone feature, but it leans on infrastructure shipped by [contribution-improvement-plan.prompt.md](contribution-improvement-plan.prompt.md): the `contributions` table, approval flow, `affected_bounds`, the TOPS-map regen worker, and the feature-flag system from Phase 0b. Land that plan's Phase 0 first; the rest of this plan can ship in parallel with Phases 1/3/4a.

---

## Decisions locked from clarification

- **Version source**: a single server-side env var `CURRENT_GAME_VERSION` (default `"1.22"`). Not read from the uploaded `.db`, not user-selectable. The server stamps every new contribution with whatever the env var says at approval time.
- **Spatial unit**: a fixed coarse grid with cells of **512 × 512 blocks** (= 16 × 16 tiles). At Vintage Story's default 1024k-block world that's ~2000 cells per axis worst-case, but in practice only the explored region is populated.
- **Visualisation**: a single tinted overlay layer on the existing TOPS map page, toggleable, with one colour per known version and a legend.
- **Truth source**: the `contributions.game_version` column. The cell grid is a derived cache, rebuilt incrementally by the TOPS regen worker. Never queried as the source of truth.
- **Cell value rule — first-write-wins**: a cell's version is set by the **first** approved contribution to touch any tile in it, and is **never** overwritten by later contributions, including region-overwrite contributions. Rationale: the version represents *when this area was generated by the game engine*. Region overwrites only refresh the satellite view (tree removals, landscaping, structures); they do not regenerate terrain, so the underlying world version is unchanged.
- **Completion tracking**: each cell additionally tracks whether all 256 of its tiles (16×16) have been mapped at least once. Completion is monotonic — once true, it stays true (a region overwrite cannot "un-complete" a cell).

## Phase 5a — Capture game version on every contribution

**Steps**

1. **Config**: add `CURRENT_GAME_VERSION: str = "1.22"` to [backend/app/config.py](backend/app/config.py). Read once per process; surfaced via `/api/admin/feature-flags` payload (read-only display) so admins can verify what's set without shelling into the server.
2. **Schema**: add column `game_version TEXT` to `contributions`, nullable for legacy rows. Backfill existing rows once with `'unknown'` via a one-off migration script in [backend/](backend/).
3. **Approval flow**: in [backend/app/routes/contribute_r2.py](backend/app/routes/contribute_r2.py), set `game_version = settings.CURRENT_GAME_VERSION` when writing the `contributions` row at approval (not at upload — version reflects when the data landed, and bumping the env var should not retroactively re-stamp pending uploads).
4. **API surface**: include `game_version` in the `/contribute/info` response for both pending and approved rows (no privacy concern — it's the server's version, not the contributor's).
5. **Frontend**: show the version next to each row on [frontend/src/pages/ContributePage.tsx](frontend/src/pages/ContributePage.tsx) ("v1.22" badge).

**Verification**
- Set `CURRENT_GAME_VERSION=1.23` → restart backend → approve a contribution → row has `game_version = "1.23"`.
- Legacy rows show `unknown` after backfill.
- Bumping the env var does not change historical rows.

## Phase 5b — Coarse-grid version map

The grid is stored as a single R2 object (one file is fine — even a fully-explored Vintage Story world has at most a few tens of thousands of populated cells, and the file is read once per page load and cached client-side).

**Cell key encoding**

- Cell coordinate `(cx, cz)` derived from world-block coords: `cx = block_x // 512`, `cz = block_z // 512`. Negative coords supported (Python `//` floors toward negative).
- Cell key = single 64-bit int: `(cz << 32) | (cx & 0xFFFFFFFF)` (signed→unsigned bit reinterpret in 32 bits each). Avoids string keys; allows packed binary storage.

**Storage format**

R2 object `tops_map/version_overlay.json` (gzip-encoded by R2):

```json
{
  "schema_version": 2,
  "cell_size_blocks": 512,
  "tiles_per_cell": 256,
  "generated_at": "2026-04-27T13:00:00Z",
  "versions": ["1.20", "1.21", "1.22", "unknown"],
  "cells": [
    [cx, cz, version_index, tile_count],
    ...
  ]
}
```

Why this shape:

- Array of tuples instead of an object map — smaller payload, gzip-friendly, decodes straight into a `Map<bigint, {version: number, tiles: number}>` on the frontend.
- `version_index` is an offset into the `versions` array so we don't repeat strings.
- `tile_count` is the number of distinct tiles ever mapped inside the cell (capped at 256). Completion = `tile_count == tiles_per_cell`. Stored as the count rather than a boolean so the frontend can render "fill-aware alpha" if we want it later — zero extra storage cost (one small int per cell).
- `version_index` for a given cell is **immutable once set**. `tile_count` is monotonically non-decreasing.

**Steps**

1. **Cell math helpers** in new [backend/app/core/version_grid.py](backend/app/core/version_grid.py):
   - `tile_to_cell(tx, tz) -> (cx, cz)`: tile coords (32-block units) → cell coords (512-block units). One cell = 16 × 16 tiles.
   - `cells_for_positions(positions: Iterable[int]) -> dict[(cx,cz), set[tile_key]]`: groups a contribution's tile positions by the cell they fall in. Used for accurate `tile_count` updates (bounds alone over-count when a contribution only fills part of a cell).
2. **Build the map** (`build_version_overlay()` in [backend/app/core/version_grid.py](backend/app/core/version_grid.py)):
   - Iterate `contributions` ordered `approved_at ASC` (oldest first), filtered to `status = 'approved'`. Reverted contributions are excluded.
   - For each contribution, read its `undo/<id>.added.bin` (the canonical list of tiles this contribution actually wrote into combined for the first time — see Phase 4b of the contribution plan). For region-overwrite contributions, the `replaced.db` rows are **not** counted toward `tile_count` (those tiles were already present from a previous contribution).
   - For each touched cell:
     - If the cell has no entry yet → create it with `version_index = this contribution's version`, `tile_count = number_of_added_tiles_in_cell`.
     - If the cell already has an entry → keep its existing `version_index` (first-write-wins); add to `tile_count`, capped at 256.
   - Return the JSON-serialisable dict above; uploader writes to R2.
3. **Incremental update path** (`update_overlay_for_contribution(contribution_id)`):
   - Triggered from the approval flow under map lock.
   - Read the contribution's `added.bin` (positions actually new to combined).
   - For each cell those positions fall in:
     - If cell missing in overlay → create with this contribution's version + count.
     - If cell present → leave `version_index` untouched; bump `tile_count` by the number of newly-added tiles in that cell, capped at 256.
   - Region-overwrite contributions: `added.bin` only contains tiles that were genuinely new in the region (gap-fills). Overwritten tiles (from `replaced.db`) are skipped — they neither change the version nor the count, by the rule above. This is exactly what we want.
   - For a revert: the version of any cell whose first-writer was the reverted contribution must roll back to the next-oldest contribution that touched it (or disappear if none). This cannot be done incrementally without a per-cell contribution stack. Schedule a **full rebuild** instead (cheap; sub-second for current map sizes). Same path handles backup-restore (Phase 4a).
4. **Background job**: hook into the existing TOPS-map regen worker in [backend/app/tasks/](backend/app/tasks/). After the chunk regen pass finishes, trigger overlay update in the same worker tick (cheap; happens under the same map lock).
5. **Endpoint**: `GET /api/tops-map/version-overlay` returns a presigned URL to the R2 object (5-min TTL, regenerated per request). Mirrors the pattern used by the existing TOPS chunk endpoints. No auth beyond a valid read key.
6. **Backfill bootstrap**: one-time admin endpoint `POST /api/admin/version-overlay/rebuild` runs `build_version_overlay()` from scratch and uploads. Required after the Phase 5a backfill so `unknown` cells appear.

**Verification**
- Approve two contributions in different cells → overlay has two entries with correct versions and accurate `tile_count`s.
- Bump `CURRENT_GAME_VERSION`, approve a third contribution that adds 10 new tiles in a cell already covered by version A → cell still reads version A; `tile_count` increases by 10.
- Approve a region-overwrite contribution under v1.23 over a cell first-written by v1.22 → cell still reads v1.22; `tile_count` unchanged (no tiles were genuinely new).
- Approve enough contributions to fill all 256 tiles of a cell → `tile_count == 256`; subsequent contributions to the same cell leave version and count untouched.
- Revert a contribution that was the first writer of cell C → full rebuild runs → cell C either inherits the next-oldest contribution's version, or disappears entirely if none.
- Restore from backup (Phase 4a) → full rebuild runs → cells past the backup point return to backup-era versions.
- Build overlay for a synthetic 1000-contribution dataset → output < 5 MB uncompressed, < 500 ms to render.

## Phase 5c — Frontend tinted overlay

**Steps**

1. **Fetch on TOPS map load**: extend the TOPS map page (likely [frontend/src/pages/](frontend/src/pages/), the page that uses `AdminResolutionPanel`) to call the new endpoint, fetch the JSON, decode into a `Map<bigint, {version: number, tiles: number}>`.
2. **Overlay rendering**: a separate canvas layer above the chunk-stitched TOPS map. For each cell present in the map, fill a 512×512-block rectangle (translated to screen coordinates) with the version's colour. Alpha is fill-aware: `alpha = 0.15 + 0.35 * (tile_count / 256)` — a sparsely-mapped cell shows as a faint tint, a fully-completed cell shows at full overlay opacity (0.5). Cells not in the map render no tint at all.
3. **Completion outline**: cells with `tile_count == 256` get a thin solid border in the version's colour, drawn on top of the fill. Quick visual signal that the area is fully mapped under that version.
4. **Colour palette**: deterministic from version string — hash the version with a small palette (8–12 distinguishable colours; reuse on collision). Stored in a constant in a new [frontend/src/lib/versionColors.ts](frontend/src/lib/versionColors.ts) so admins can curate exact colours later. Default for `unknown` is a neutral grey.
5. **Toggle UI**: a checkbox in the TOPS map controls bar — "Show version overlay". Default off (don't surprise users on first load). State persisted in `localStorage`.
6. **Legend**: when overlay is on, show a floating legend in a corner listing each version present in the visible viewport with its colour swatch, cell count, and completed-cell count (e.g. "v1.22 — 142 cells (37 fully mapped)").
7. **Click for detail**: clicking a cell while the overlay is on opens a small popover: "v1.22 — first mapped by [contributor name] on [approved_at] — N / 256 tiles mapped". Requires a separate lookup endpoint `GET /api/tops-map/cell-info?cx=..&cz=..` returning the **first-writer** contribution for the cell plus the current `tile_count`. Query: `SELECT contributor, approved_at, game_version FROM contributions WHERE status = 'approved' AND affected_bounds intersects cell ORDER BY approved_at ASC LIMIT 1` (note the ASC — we want the oldest, not the newest).
8. **Feature flag**: the whole overlay (endpoint + UI) gated by `is_feature_enabled('version_overlay')` (add to the seed list in Phase 0b of the contribution plan).

**Verification**
- Toggle overlay on with two versions present → two distinct tints visible, legend matches counts.
- A sparsely-mapped cell renders fainter than a fully-mapped one of the same version.
- A fully-mapped cell (`tile_count == 256`) shows the completion outline.
- Pan the map → overlay layer scrolls in sync with the base layer.
- Click a tinted cell → popover shows the original (first-writer) contributor and version, plus current N/256 tile count.
- Toggle off → overlay disappears, base map unchanged.
- Disable feature flag → overlay endpoint returns 404, UI checkbox disappears.

**Out of scope**

- Per-tile version (stays at 512-block cell granularity).
- Reading game version from the uploaded `.db` (locked to env var only).
- User-selectable version on upload.
- Mid-cell version transitions — once a cell has a version, it keeps it forever (by design).
- "Mixed version" cells — not possible under first-write-wins.
- Version-aware filtering in the contribute history (could come later).
- Migrating away from JSON to a tile pyramid (only worth it if cell count exceeds ~100k).
- Showing version diffs over time (animation / time-slider).

**Phase order**

1. Phase 5a (env var + column + stamp on approval) — fully independent, can ship immediately after Phase 0 of the contribution plan.
2. Phase 5b (overlay build + endpoint) — depends on 5a having data to read; can ship without 5c.
3. Phase 5c (frontend tinted overlay) — depends on 5b's endpoint.

**Further considerations**

1. **Version string normalisation** — `1.22`, `1.22.0`, `1.22-rc.1` should probably collapse for display. For now treat the env var as opaque; revisit when you actually run a release-candidate phase.
2. **Cell-info popover cost** — every click hits Supabase. If the TOPS map page becomes high-traffic, cache the cell-info responses in Redis or in memory keyed by `(cx, cz, overlay_generated_at)`.
3. **Overlay file growth** — JSON is fine until ~100k cells. Beyond that, switch to a packed binary (one record = `int32 cx + int32 cz + uint8 version_index`) and decode in JS with `DataView`. Same R2 path, different content-type. Easy migration via `schema_version` field.
4. **What happens to cells when a contribution is reverted but it was the only contribution in that cell?** — full rebuild correctly drops the cell (no entry in the map → no tint). Frontend handles "missing cell" as "no version known". Confirmed by the Phase 5b verification step.
5. **Why `tile_count` doesn't decrement on revert** — it actually does, via the full rebuild path. We deliberately don't try to incrementally decrement on revert: keeping the increment path one-way avoids the bookkeeping complexity of "who first-wrote this tile inside this cell" beyond the contribution level. Full rebuild is correct and currently sub-second; revisit if that changes.
