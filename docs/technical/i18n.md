# Multi-language support (i18n)

Technical reference for adding and using translations in the frontend. The
project ships a small, dependency-free i18n layer under
[frontend/src/lib/i18n/](../../frontend/src/lib/i18n) that provides type-safe
keys, plural rules, runtime interpolation, and a React `Trans` component for
inline markup.

## Architecture overview

| File | Purpose |
| --- | --- |
| [frontend/src/lib/i18n/types.ts](../../frontend/src/lib/i18n/types.ts) | Type machinery (`PathOf`, `ArgsForPath`, `PluralEntry`, …). The English dictionary type drives autocompletion across the app. |
| [frontend/src/lib/i18n/registry.ts](../../frontend/src/lib/i18n/registry.ts) | Locale list, default locale, lazy loaders, and the fallback dictionary. |
| [frontend/src/lib/i18n/locales/en.ts](../../frontend/src/lib/i18n/locales/en.ts) | Source-of-truth English dictionary. Its inferred type is `TranslationSchema`. |
| [frontend/src/lib/i18n/locales/ru.ts](../../frontend/src/lib/i18n/locales/ru.ts) | Russian translations as a `DeepPartial<TranslationSchema>` — missing keys fall back to English. |
| [frontend/src/lib/i18n/format.ts](../../frontend/src/lib/i18n/format.ts) | `getByPath`, `interpolate`, and `Intl.PluralRules`-based plural selection. |
| [frontend/src/lib/i18n/core.ts](../../frontend/src/lib/i18n/core.ts) | `translate()` — picks the entry, applies plural rules, and falls back to English with a dev-only console warning when a key is missing. |
| [frontend/src/lib/i18n/react.tsx](../../frontend/src/lib/i18n/react.tsx) | `<I18nProvider>`, `useTranslation`, `useFormat`, and `<Trans>`. |
| [frontend/src/lib/i18n/index.ts](../../frontend/src/lib/i18n/index.ts) | Public barrel — always import from `@/lib/i18n`. |
| [frontend/src/components/LanguageSwitcher.tsx](../../frontend/src/components/LanguageSwitcher.tsx) | UI control that calls `setLocale`. |
| [frontend/src/store/slices/i18n.ts](../../frontend/src/store/slices/i18n.ts) | Redux slice that persists the selected locale. |

Runtime flow:

1. `<I18nProvider>` reads `locale` from Redux and lazily imports the matching
   dictionary via `LOCALE_LOADERS[locale]()`.
2. The loaded dictionary is cached in a module-level `Map`.
3. `t(path, args)` and `<Trans path=… />` call `translate()`, which walks the
   dictionary by dotted path, picks the correct plural form when needed, and
   interpolates `{var}` tokens.
4. If the key (or a required sub-form) is missing in the active dictionary, it
   falls back to English and logs `[i18n] missing key …` in development.

## Adding or editing a translation key

The English file is the schema. Every other locale is typed as
`DeepPartial<WidenLeafStrings<TranslationSchema>>`, so a missing key is allowed
but a typo is a compile error.

1. Open [frontend/src/lib/i18n/locales/en.ts](../../frontend/src/lib/i18n/locales/en.ts)
   and add the key to the appropriate nested namespace:

    ```ts
    // frontend/src/lib/i18n/locales/en.ts
    export const en = {
      account: {
        page: {
          title: "Your account",
          welcome: "Welcome back, {name}.",
        },
      },
      // …
    } as const;
    ```

2. Optionally add the same key to
   [frontend/src/lib/i18n/locales/ru.ts](../../frontend/src/lib/i18n/locales/ru.ts).
   If you skip this step, Russian users see the English text and a dev-only
   warning is emitted.

3. Use it from a component:

    ```tsx
    import { useTranslation } from "@/lib/i18n";

    export function AccountHeader({ name }: { name: string }) {
      const { t } = useTranslation();
      return (
        <header>
          <h1>{t("account.page.title")}</h1>
          <p>{t("account.page.welcome", { name })}</p>
        </header>
      );
    }
    ```

The dotted path `"account.page.title"` is checked against `TranslationSchema`,
so renaming `title` in `en.ts` surfaces every stale call site as a TypeScript
error.

## `useTranslation()` and the `t()` function

```ts
const { t, locale, setLocale } = useTranslation();
```

- `t(path)` — returns a `string` for keys with no variables.
- `t(path, args)` — returns a `string`. The shape of `args` is inferred from
  the English entry. Passing extra or missing keys is a type error.
- `locale` — the active locale code (`"en" | "ru"`).
- `setLocale(next)` — dispatches the Redux action and triggers a dictionary
  load.

`t()` is the right tool whenever the result is plain text: button labels,
`aria-label`s, `title` attributes, toast messages, `placeholder`s, `alt` text,
etc. If you need React nodes inside the translation, reach for
[`Trans`](#the-trans-component-rich-text-with-react-elements) instead.

```tsx
<button aria-label={t("common.selectLanguage")}>…</button>
```

`useTranslation()` must be called inside `<I18nProvider>` (mounted near the
app root); otherwise it throws.

## Interpolated variables

Variables use single-brace tokens inside the dictionary string:

```ts
// en.ts
common: {
  copyPathForLabel: "Copy {label} path",
},
```

```tsx
const { t } = useTranslation();
t("common.copyPathForLabel", { label: "Windows" });
// → "Copy Windows path"
```

Rules and behavior:

- Token names must match `[a-zA-Z0-9_]+`.
- Values are `string | number` (`PrimitiveValue`). Numbers are coerced with
  `String(value)`; use [`useFormat`](#number-date-and-time-formatting-useformat)
  if you need locale-aware number formatting.
- A token with no matching argument renders literally as `{label}` so that
  missing data is visible in the UI and in tests.
- The variable name and type are extracted from the English string by
  `ExtractVars` / `ArgsForPath`, so the compiler enforces that you pass the
  right keys. Adding a new `{var}` token to the English string is a breaking
  change for every call site.

Multiple variables work the same way:

```ts
mapDbFileHelp: {
  fileSize: "{name} ({size})",
},
```

```tsx
t("mapDbFileHelp.fileSize", { name: "world.db", size: "12 MB" });
```

## Plurals

A plural entry is an object with `other` (required) plus any of `one`, `few`,
`many`. Selection uses
[`Intl.PluralRules`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Global_Objects/Intl/PluralRules)
for the active locale, so authors don't need to encode language-specific
rules.

```ts
// en.ts
contributeTls: {
  pending: {
    one:   "{count} pending review",
    other: "{count} pending reviews",
  },
},
```

```ts
// ru.ts — Russian has one/few/many/other
contributeTls: {
  pending: {
    one:   "{count} ожидает проверки",
    few:   "{count} ожидают проверки",
    many:  "{count} ожидают проверки",
    other: "{count} ожидают проверки",
  },
},
```

```tsx
t("contributeTls.pending", { count: items.length });
```

Requirements:

- The args object **must** include `count: number`. The type machinery enforces
  this for plural entries.
- The English entry must define `other`. Other forms are optional.
- If the active locale's entry is missing a form that `Intl.PluralRules`
  selected, the code falls back to `other`, then `one`, then `""`.
- The `{count}` token still needs to be written explicitly in each form — it
  is not auto-substituted.

## The `Trans` component (rich text with React elements)

`t()` returns a string, so it can't embed React elements (`<strong>`, links,
`<code>` blocks, …) without HTML injection. `Trans` solves this with named
component slots that map to placeholder tags in the translation string.

Dictionary entries use ordinary angle-bracket tags whose names are *local* to
that entry — they have no relationship to real HTML elements:

```ts
// en.ts
generalPage: {
  disclaimer:
    "This is an <strong>unofficial</strong> tool. Read the <em>terms</em> first.",
  cairnDefinition:
    "A <strong>cairn</strong> is a <link>man-made stack of stones</link>, often used as a <noun>landmark</noun>.",
},
```

Render:

```tsx
import { Trans } from "@/lib/i18n";

<Trans
  path="generalPage.disclaimer"
  components={{
    strong: <strong />,
    em: <em />,
  }}
/>

<Trans
  path="generalPage.cairnDefinition"
  components={{
    strong: <strong className="not-italic text-foreground" />,
    link: (
      <a
        href="https://en.wikipedia.org/wiki/Cairn"
        target="_blank"
        rel="noopener noreferrer"
        className="underline decoration-dotted underline-offset-2"
      />
    ),
    noun: <span className="not-italic" />,
  }}
/>
```

Behavior:

- Tag names in `components` must match the tags used in the dictionary entry.
  Unmatched tags render as their inner text (no wrapper element).
- Components are passed as **elements** (e.g. `<a href="…" />`), not types.
  `Trans` clones the element and injects translated `children`; any other
  props on the element (`href`, `className`, `target`, …) are preserved.
- Tags can nest and can be reused multiple times. Each occurrence becomes its
  own React element with a generated `key`.
- `{var}` tokens still work inside the wrapped content:

    ```ts
    welcome: "Hi <strong>{name}</strong>, you have <link>{count} messages</link>.",
    ```

    ```tsx
    <Trans
      path="inbox.welcome"
      values={{ name, count }}
      components={{ strong: <strong />, link: <a href="/inbox" /> }}
    />
    ```

- `values` is typed the same way as `t()`'s args (extracted from the English
  entry), so the compiler still enforces the variable names.
- `Trans` returns a `ReactNode`. To embed it inside larger paragraphs, wrap it
  in the appropriate block element yourself — see
  [GeneralPage.tsx](../../frontend/src/pages/GeneralPage.tsx) and
  [MapDbFileHelp.tsx](../../frontend/src/components/MapDbFileHelp.tsx) for
  representative call sites.

### When to choose `t()` vs `Trans`

| Situation | Use |
| --- | --- |
| Plain text, no markup | `t()` |
| Attributes (`aria-label`, `title`, `placeholder`, `alt`) | `t()` |
| Toast / alert messages without inline elements | `t()` |
| Pluralized counts shown as text | `t()` |
| Sentence that needs `<strong>`, `<a>`, `<code>`, icons, etc. | `Trans` |
| Sentence with embedded React elements *and* `{var}` substitutions | `Trans` with `values` and `components` |

Never assemble translated sentences by concatenating multiple `t()` calls
around JSX — word order varies between languages and that approach can't be
translated correctly. Use a single `Trans` entry instead.

## Number, date, and time formatting (`useFormat`)

`useFormat()` returns helpers bound to the current locale's `intlCode` (the
BCP-47 tag stored in `LOCALE_META`):

```tsx
import { useFormat } from "@/lib/i18n";

function PostMeta({ publishedAt, views }: { publishedAt: string; views: number }) {
  const { dateTime, number } = useFormat();
  return (
    <p>
      {dateTime(publishedAt, { dateStyle: "medium" })} · {number(views)} views
    </p>
  );
}
```

- `number(value, options?)` wraps `Intl.NumberFormat`.
- `dateTime(value, options?)` wraps `Intl.DateTimeFormat`. `value` may be a
  `Date`, an ISO string, or a numeric timestamp.
- `locale` and `intlCode` are also returned for ad-hoc `Intl.*` calls.

See [BlogIndexPage.tsx](../../frontend/src/pages/blog/BlogIndexPage.tsx) and
[IdentifyMapsResult.tsx](../../frontend/src/components/identify-maps/IdentifyMapsResult.tsx)
for live examples.

## Switching locale at runtime

`useTranslation` exposes `setLocale(next)`, which dispatches the Redux action
in [frontend/src/store/slices/i18n.ts](../../frontend/src/store/slices/i18n.ts).
The provider reacts by lazy-loading the new dictionary, updating
`document.documentElement.lang`, and re-rendering the tree.

The shared UI control is
[`<LanguageSwitcher />`](../../frontend/src/components/LanguageSwitcher.tsx) —
prefer mounting it rather than building a custom selector.

## Adding a new locale

1. Add the code to the `Locale` union in
   [types.ts](../../frontend/src/lib/i18n/types.ts):

    ```ts
    export type Locale = "en" | "ru" | "de";
    ```

2. Register metadata and a loader in
   [registry.ts](../../frontend/src/lib/i18n/registry.ts):

    ```ts
    export const LOCALE_META: Record<Locale, LocaleMeta> = {
      en: { code: "en", label: "English", nativeLabel: "English",  intlCode: "en" },
      ru: { code: "ru", label: "Russian", nativeLabel: "Русский",  intlCode: "ru" },
      de: { code: "de", label: "German",  nativeLabel: "Deutsch",  intlCode: "de" },
    };

    export const LOCALE_LOADERS: Record<Locale, () => Promise<Dict>> = {
      en: async () => en,
      ru: async () => (await import("./locales/ru")).default,
      de: async () => (await import("./locales/de")).default,
    };
    ```

3. Create `frontend/src/lib/i18n/locales/de.ts` exporting a default
   `DeepPartial<WidenLeafStrings<TranslationSchema>>`:

    ```ts
    import type { DeepPartial, TranslationSchema, WidenLeafStrings } from "../types";

    const de: DeepPartial<WidenLeafStrings<TranslationSchema>> = {
      common: { language: "Sprache" },
      // …
    };

    export default de;
    ```

4. Add the new code to the `OPTIONS` array in
   [LanguageSwitcher.tsx](../../frontend/src/components/LanguageSwitcher.tsx).

Anything you don't translate falls back to English automatically.

## Conventions and gotchas

- **Always import from `@/lib/i18n`.** The barrel re-exports the public
  surface; do not import from `./react` or `./core` directly.
- **Keep keys in the namespace they belong to.** Page-scoped keys live under
  `<page>.…`; cross-cutting strings live under `common.…`. New top-level
  namespaces should be intentional.
- **Never build strings by concatenation across `t()` calls.** Put the full
  sentence in the dictionary and use `{vars}` or `<Trans>` slots.
- **Plural strings must keep `{count}` in every form.** It's not implicit.
- **Tag names in `<Trans>` are author-defined.** They don't have to match real
  HTML; pick names that read well in the translation (`<link>`, `<strong>`,
  `<code>`, `<note>`, …) and map them to elements in `components`.
- **Renaming or removing English keys is a breaking change** — TypeScript will
  flag every call site. Run `tsc` after editing `en.ts`.
- **Missing keys are not fatal.** They fall back to English at runtime and log
  `[i18n] missing key "<path>" in locale "<locale>"` once per key in `DEV`.
  Treat those warnings as TODOs for the localization pass.
- **`<I18nProvider>` is mounted near the app root** — components rendered
  outside it (e.g. error boundaries that render before providers mount) cannot
  use `useTranslation`.
