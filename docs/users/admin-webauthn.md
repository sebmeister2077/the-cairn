# Admin WebAuthn (Passkey 2FA)

Admin API keys grant unrestricted access to the system. Because they live in
plaintext on the operator's laptop and travel as a single `X-API-Key` header,
a leaked admin key is catastrophic. Phase 4c adds a **WebAuthn-backed second
factor** so that an attacker holding only the API key cannot reach
`/api/admin/*` from a different device.

The factor is a **passkey** — a public-private keypair generated by the
admin's OS / browser (Windows Hello, Touch ID, Android biometrics, a YubiKey
or other FIDO2 authenticator, or a synced credential from a password
manager). The private key never leaves the device. The server only stores
the public key and a monotonic signature counter.

## Threat model

| Attack | Mitigated? |
|---|---|
| API key leaked from operator's clipboard / repo / log | ✅ — attacker has no passkey |
| API key phished onto another device | ✅ — passkey is bound to the device + origin |
| Replayed admin request with stolen `X-Admin-Session` token | ⚠️ Partial — token is bound to client IP hash; rotates every 8 h; restarts of the API server invalidate all tokens |
| Compromised admin browser (token + key both stolen) | ❌ — equivalent to a full session takeover |
| Lost / destroyed device with the only passkey | ⚠️ Recoverable by direct DB access (see [Recovery](#recovery)) |

## Configuration

Set these env vars on the API host (Render):

| Variable | Default | Purpose |
|---|---|---|
| `WEBAUTHN_RP_ID` | _unset_ | The Relying Party ID. **Must match the public hostname** of the frontend, e.g. `vs-waypoints.example.com`. Without this, all WebAuthn endpoints return `503 webauthn_not_configured`. |
| `WEBAUTHN_RP_NAME` | `VS Waypoints Admin` | Human-readable name shown in the OS prompt. |
| `WEBAUTHN_ORIGINS` | falls back to `ALLOWED_ORIGINS` | Comma-separated allow-list of `https://...` origins permitted to register / assert. Must include the production frontend URL. |
| `WEBAUTHN_SESSION_TTL_SECONDS` | `28800` (8 h) | Lifetime of the `X-Admin-Session` token issued after a successful assertion. |
| `WEBAUTHN_ENFORCE` | `True` | If `True`, admins with at least one registered credential **must** present an `X-Admin-Session`. If `False`, passkey verification is purely optional (use during initial rollout). |

> ⚠️ **Local dev tip.** `localhost` is the only non-HTTPS origin browsers
> accept for WebAuthn. If you test from `http://localhost:5173`, set
> `WEBAUTHN_RP_ID=localhost` and add `http://localhost:5173` to
> `WEBAUTHN_ORIGINS`.

## Storage

A new table is created automatically by `database._run_migrations`:

```sql
CREATE TABLE webauthn_credentials (
    id BIGSERIAL PRIMARY KEY,
    api_key TEXT NOT NULL,
    name TEXT NOT NULL,
    credential_id BYTEA NOT NULL UNIQUE,
    public_key BYTEA NOT NULL,
    sign_count BIGINT NOT NULL DEFAULT 0,
    transports TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    last_used_at TIMESTAMPTZ
);
CREATE INDEX idx_webauthn_api_key ON webauthn_credentials (api_key);
```

Notes:
- `credential_id` and `public_key` are stored as raw bytes (the spec types).
  Helper functions in `core/database.py` decode the `BYTEA` ↔ `bytes` round-trip.
- `api_key` here is the literal admin key string (the same value the admin
  pastes on the frontend). Multiple keys → independent credential sets.
- `sign_count` is updated on every assertion and used to detect cloned
  authenticators (per WebAuthn §6.1.1).
- Private keys **are never stored** anywhere on the server.

## Session tokens

After a successful assertion the server returns:

```json
{ "session_token": "…opaque random base64…", "expires_in": 28800 }
```

The frontend persists it in `localStorage` under `admin_session` and sends
it on every admin request as `X-Admin-Session: <token>`.

The token store is **in-memory only** (`_admin_sessions: dict` in
`backend/app/auth.py`). Each entry records:

- the issuing `api_key`,
- the SHA-256 IP hash of the issuing client (using `IP_HASH_SALT`),
- the absolute expiry timestamp.

Implications:

- Restarting the API server invalidates every admin session — admins must
  re-tap their passkey. This is intentional: it bounds the blast radius of
  a stolen token to the current process lifetime.
- Tokens are bound to the client IP hash. Switching networks (phone hotspot
  → home wifi) forces re-verification.
- The frontend automatically clears the token whenever the API key changes.

## Endpoints

All paths are prefixed with `/api`. Status / list / register / assert /
delete / logout share `Depends(require_admin_keyonly)` — i.e. they accept
a valid admin API key without requiring a pre-existing passkey session.
This breaks the bootstrap chicken-and-egg of "you need a session to
register the credential that mints the session".

| Method | Path | Description |
|---|---|---|
| `GET` | `/admin/webauthn/status` | `{ configured, enrolled, enforced, session_ttl_seconds }`. Used by the frontend to decide whether to prompt. |
| `GET` | `/admin/webauthn/credentials` | List credentials owned by the calling admin. |
| `DELETE` | `/admin/webauthn/credentials/{id}` | Revoke one credential. **Requires** a current `X-Admin-Session` if any credentials exist (so a stolen API key cannot wipe enrolment). |
| `POST` | `/admin/webauthn/register/begin` | Body `{name}`. Returns `{options}` for `navigator.credentials.create()`. |
| `POST` | `/admin/webauthn/register/complete` | Body `{name, credential}`. Verifies attestation and persists the credential. |
| `POST` | `/admin/webauthn/auth/begin` | Returns `{options}` for `navigator.credentials.get()`. The challenge expires in 5 min. |
| `POST` | `/admin/webauthn/auth/complete` | Body `{credential}`. Verifies the assertion, bumps `sign_count`, mints `{session_token, expires_in}`. |
| `POST` | `/admin/webauthn/logout` | Revokes the current session token server-side. |

## Frontend flow

1. Admin pastes their API key in the **API key dialog**.
2. After the key is accepted, the dialog calls `/admin/webauthn/status`.
   - `enrolled=true` → opens the passkey dialog in **`assert` mode**, marked
     `required`. The "Not now" button is hidden and Esc / overlay click are
     disabled — the only ways out are completing the ceremony or closing the
     tab. Reloading the page brings the same modal back (see step 3). Any
     stale session token is cleared first so the server forces a fresh
     ceremony.
   - `enrolled=false, configured=true` → opens the dialog in **`register`
     mode** as a soft prompt. The admin **can press "Not now"** to dismiss
     it: no session token is set (none is needed yet — the server only
     enforces `_enforce_passkey_session` when the admin has at least one
     credential), so all `/api/admin/*` endpoints continue to work with just
     the API key. The same soft prompt re-appears on every subsequent API
     key save and on every reload until the admin enrols their first
     passkey or removes the API key. Admins can also enrol any time from the
     **Admin passkeys** card on the Account page.
3. On every page reload, the App root checks: if `is_admin && !adminSession`,
   it re-fetches `/admin/webauthn/status` and re-prompts using the same
   rules as step 2 (hard gate if enrolled, soft prompt if not).
4. The Account page exposes an **Admin passkeys** card to add / remove
   credentials, view the current session expiry, and sign out of the
   passkey session without dropping the API key.

The browser-side ceremony uses `@simplewebauthn/browser` for safe
encoding / decoding of the binary fields.

## Audit log entries

Every passkey lifecycle event is recorded via
`accounts_db.audit_log(actor=api_key, ...)`:

| Event | When |
|---|---|
| `webauthn.credential_registered` | After `register/complete` succeeds. |
| `webauthn.signed_in` | After `auth/complete` mints a session. |
| `webauthn.credential_deleted` | After `DELETE /credentials/{id}`. |

## Recovery

**Lost the only device with the passkey?** The fallback is direct DB
access (the API has no "send recovery email" — admins are expected to
have privileged DB access anyway):

```sql
-- Wipe the admin's enrolment so the next login skips the assert step
DELETE FROM webauthn_credentials WHERE api_key = '<admin api key>';
```

Then log in normally; the dialog will offer to register a new credential.

**Trade-off.** This means anyone with DB access can defeat the second
factor. The mitigation is operational: keep DB credentials in a secrets
manager, audit access, and prefer rotating both the API key and the
passkey if you suspect the DB has been touched.

## Disabling

To turn the gate off entirely (e.g. during an incident response window):

1. Set `WEBAUTHN_ENFORCE=False` and redeploy. Admins can still use
   passkeys voluntarily, but `require_admin` no longer rejects requests
   that lack a session.
2. Or unset `WEBAUTHN_RP_ID`. All `/admin/webauthn/*` endpoints return
   `503 webauthn_not_configured` and the frontend silently hides the
   card.
